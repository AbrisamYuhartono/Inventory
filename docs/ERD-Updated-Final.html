🔄 Perubahan Utama: Pemisahan Users dan Pegawai

1. Entitas Users (Sistem Login)

• Entitas terpisah untuk manajemen akses sistem
• Berisi username, password, role (Superadmin/Admin/User)
• Hanya dapat dikelola oleh Admin (untuk User) dan Superadmin (untuk semua)
• Relasi optional (0:1) dengan Pegawai untuk role User

2. Entitas Pegawai (Data Pegawai)

• Entitas terpisah untuk data pegawai lengkap
• Berisi NIP, KARPEG, pangkat_golongan, jabatan, unit
• Data pegawai dapat dilihat berdasarkan unit masing-masing
• Unit: Setditjen ILMATE, Logam, IPAMP, IMATAB, IET

3. Entitas Items (Diperbaharui)

• Menambahkan field brand, model, description
• Memperjelas informasi barang untuk kemudahan identifikasi
• QR Code otomatis untuk setiap barang

4. Entitas Lending_Requests & Repair_Requests

• Mengganti nama dari Lending_Records menjadi Lending_Requests
• Menambahkan field requested_by untuk tracking user yang mengajukan
• Relasi dengan Pegawai untuk data peminjam/pengaju
• Workflow: User pilih barang → pilih pegawai → submit request

5. Akses Data Berdasarkan Unit

• User hanya dapat melihat pegawai dari unit yang sama
• Admin dan Superadmin dapat melihat semua unit
• Pemisahan data untuk keamanan dan privasi

Detailed Relationship Mapping

Users → Pegawai (0:1)

User dengan role 'User' dapat terhubung dengan satu pegawai. Admin dan Superadmin tidak perlu terhubung dengan pegawai.

Users → Lending_Requests (1:N)

Satu user dapat mengajukan banyak permintaan peminjaman. Setiap permintaan dicatat siapa yang mengajukan.

Users → Repair_Requests (1:N)

Satu user dapat mengajukan banyak permintaan perbaikan. Tracking lengkap siapa yang mengajukan permintaan.

Pegawai → Lending_Requests (1:N)

Satu pegawai dapat menjadi peminjam di banyak permintaan. Data pegawai lengkap tersimpan untuk dokumentasi.

Pegawai → Repair_Requests (1:N)

Satu pegawai dapat mengajukan banyak permintaan perbaikan. Identitas pengaju tercatat lengkap.

Rooms → Items (1:N)

Satu ruangan dapat menyimpan banyak item. Setiap item berada di satu ruangan dengan PIC yang jelas.

Items → Lending_Requests (1:N)

Satu item dapat memiliki banyak permintaan peminjaman (history). Tracking lengkap penggunaan barang.

Items → Repair_Requests (1:N)

Satu item dapat memiliki banyak permintaan perbaikan (history). Maintenance record yang komprehensif.

Lending_Requests → Lending_Documents (1:N)

Satu permintaan peminjaman dapat menghasilkan banyak dokumen (PDF, kontrak, dll).
